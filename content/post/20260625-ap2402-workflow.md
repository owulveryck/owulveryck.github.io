## AP2402

### Architecture Advisor Flow

How do you delegate architecture decisions to an autonomous AI agent — and ensure fair pricing, payment guarantees, and verifiable proof?

AP2402 demonstrates a complete flow where agents delegate to other agents using the A2A protocol, with machine-to-machine payments enforced by AP2 and the HTTP 402 toll gate.

{{< scrollytelling-step phase="1" id="phase-1" >}}

## 1. Application design request

A developer within a stream-aligned team needs to design a new feature: sending 50,000 transactional emails per day.

The human asks Winston, the team's AI agent, for help. Winston understands the business context and begins evaluating the request — but this decision has enterprise-wide implications.

Winston determines that a systemic architecture validation is needed before proceeding.

{{< /scrollytelling-step >}}

{{< scrollytelling-step phase="2" id="phase-2" >}}

## 2. Architect consultation + 402

<span class="badge badge-a2a">A2A</span>
<span class="badge badge-402">402</span>

Winston delegates the request to the enterprise architect agent via the A2A protocol. This is not a simple tool call — it's a true agent-to-agent delegation with structured intent.

The architect evaluates the complexity and responds with a quote: **5 credits** for this consultation. Along with the quote, it returns a **402 Payment Required** status — the toll gate.

The architect refuses to proceed until payment is proven. No free-riding.

{{< /scrollytelling-step >}}

{{< scrollytelling-step phase="3" id="phase-3" >}}

## 3. Payment escalation

Winston cannot authorize a payment autonomously. Financial decisions require explicit human approval — this is a deliberate design choice for GDPR compliance and governance.

Winston escalates the payment request back to the human, presenting the architect's quote: 5 credits for the architecture consultation.

The human reviews, approves, and Winston proceeds to the payment flow.

{{< /scrollytelling-step >}}

{{< scrollytelling-step phase="4" id="phase-4" >}}

## 4. AP2 mandates

<span class="badge badge-ap2">AP2</span>

Winston initiates the AP2 payment protocol. First, it creates a **checkout mandate** (the cart authorization) and closes it. Then it creates a **payment mandate**, which triggers a hold of 5 credits on the buyer's account.

The payment service issues an HMAC-signed token — a portable cryptographic proof of payment containing the amount, parties, and a unique `payment_ctx_id`.

Winston resubmits the architecture request with the mandate IDs and payment token. The architect verifies the mandates via the payment broker before proceeding. The buyer creates credentials, not the seller — preventing any forgery.

{{< /scrollytelling-step >}}

{{< scrollytelling-step phase="5" id="phase-5" >}}

## 5. Design dialogue

<span class="badge badge-a2a">A2A</span>

Payment verified, the architect begins the actual work. This is a multi-turn A2A conversation, not a single request/response.

The architect asks clarifying questions: *"What's the exact daily volume? Transactional or marketing? Any regulatory constraints?"*

Winston answers with the business context. The architect iterates, refining its understanding before making a recommendation.

{{< /scrollytelling-step >}}

{{< scrollytelling-step phase="6" id="phase-6" >}}

## 6. Domain consultation

<span class="badge badge-a2a">A2A</span>

The architect doesn't hard-code feasibility logic. Instead, it delegates a technical feasibility query to **Winston@domain**, the domain expert agent for the Notifications bounded context.

The domain agent confirms: *"50,000 emails/day is feasible, but requires a quota increase and template validation."*

This is domain-driven design in action: the domain owner answers feasibility, the architect makes the systemic decision.

{{< /scrollytelling-step >}}

{{< scrollytelling-step phase="7" id="phase-7" >}}

## 7. Decision + ADR + Settlement

<span class="badge badge-ap2">AP2</span>
<span class="badge badge-a2a">A2A</span>

The architect returns a complete decision artifact: use the internal Notifications service, with concrete API specs, auth method, prerequisites, and limits.

Winston writes an **ADR** (Architecture Decision Record) containing the decision hash (SHA-256) and the `payment_ctx_id`. A deterministic validation harness verifies cryptographically that the decision is authentic and the payment was consumed — no trust in the LLM required.

The architect settles with the payment service: the actual amount used is debited, any surplus is refunded to the buyer's account.

{{< /scrollytelling-step >}}

{{< scrollytelling-step phase="8" id="conclusion" >}}

## The full picture

**A2A** enables true agent-to-agent delegation — not tool calls, but autonomous conversations with structured intent.

**AP2 + 402** ensure fair pricing: agents refuse to work until paid, mandates provide portable proof, and a neutral broker settles accounts.

**ADRs + crypto proofs** make every decision auditable and deterministically verifiable — from the initial request to the final settlement.
